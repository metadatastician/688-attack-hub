== Hard Rules

* Do not call anything `stable`, `v1.0.0`, or full release unless the stable
  release gate has been passed end to end.
* Do not publish implementation-facing work below `B` in CRG unless the work is
  genuinely abstract and makes no implementation-readiness claim.
* `D` requires RSR compliance or a documented equivalent repository discipline.
* `C` requires deep code and folder annotation, not just local confidence.
* `B` means `beta-stable`: external breadth and safe broad trial, not merely
  public visibility.
* Papers, whitepapers, release notes, and READMEs must not outrun the proofs,
  tests, or artefacts that support their claims.
* Release paths must not ship with placeholders, stubs, `FIXME`, `XXX`,
  template residue, fake fuzz, fake benches, or partial proof debt hidden as
  if it were complete.

== This repository specifically

What is verified, what is merely asserted, and what is explicitly *not*
claimed about 688 Attack Hub in particular.

=== Verified (grep-checked, CI-enforced)

* The game makes zero network calls — no `fetch`, `XMLHttpRequest`,
  `WebSocket` — and contains no `eval`/`new Function` dynamic-code
  execution. Checked by hand at repo creation and re-checked on every push
  by `ci.yml`'s `integrity` job.
* `688-attack-hub.html` is present, non-empty, and structurally valid HTML
  (`<!DOCTYPE html>` ... `</html>`).
* No `.a2ml` file in the tree is missing an SPDX header.
* No unsubstituted doubled-brace template tokens survive anywhere in the
  tree (`ci.yml`'s placeholder check).
* Every GitHub Actions `uses:` is pinned to a full 40-character commit SHA.

=== Asserted, not mechanically checked

* That the game is "original fictional content" and not derived from the
  1997 commercial game "688(I) Hunter/Killer" / "688 Attack Sub". This was a
  one-time manual check at repo creation (no copyright strings, asset names,
  or code from that game appear in the file) — it is not re-verified by CI,
  because there's nothing to grep for that would meaningfully strengthen the
  claim beyond the initial read.
* That the internet-exchange-point names and coordinates in `REGIONS` are
  drawn from real public IXP data (LINX, DE-CIX, AMS-IX, etc. at their
  approximate real coordinates) — this is flavour accuracy, not a functional
  claim, and isn't tested.
* That the countermeasure escalation (`CMS`) and upgrade balance produce a
  "fun" or "balanced" experience. This is a design/taste judgement, not
  something this repo tries to prove.
* That the orthographic globe projection is *cartographically* correct. What
  was actually done is narrower, and is recorded here rather than overstated:
  the projection was exercised in a real browser and four specific properties
  were measured — trans-Pacific edges emit two path subpaths at opposite
  screen edges (so no backbone run is drawn the long way across Eurasia);
  every far-side node reports `pointer-events: none` and `elementFromPoint`
  at its coordinates returns the land polygon rather than the node; one
  geometry pass costs 0.34 ms against a 16.7 ms frame budget; and the page
  loads with no console errors. Those four are evidence. "The globe is
  correct" is not a claim this repo makes, and there is no automated visual
  or regression test for any of it — the checks above were run once, by hand.
* A coastline polygon straddling the limb is closed along the chord between
  its horizon crossings rather than along the limb arc. This is a known and
  deliberate imperfection, visible at the edge of the disc; see
  `ARCHITECTURE.md`.
* That the globe *auto*-rotates. The geometry was swept numerically through a
  full turn (72 angles, 5 degrees apart, zero faults) and the loop's body was
  driven by hand, but the `requestAnimationFrame` scheduler itself was never
  observed firing: the test browser sets `prefers-reduced-motion`, which this
  code honours by not scheduling it, and the automated tab was backgrounded,
  where rAF does not fire at all. Those three lines of rAF recursion were
  read, not run.
+
The *manual* controls, by contrast, were exercised: a synthetic 90px drag
moved the globe, arrow keys stepped it 5 degrees (15 with Shift), and the
drag-versus-click guard was tested in all three states — a plain click on a
city registers, the click delivered after a drag is suppressed, and clicking
works again once the guard clears. Both were confirmed working with
`prefers-reduced-motion` active and no animation loop running, which is the
case that matters. See `AFFIRMATION.adoc`.

=== Explicitly NOT claimed

* **No formal verification ledger.** Unlike `f19-stealth-glider` (this org's
  other single-file browser game, which has an 11-claim verification ledger
  backed by runnable proof scripts), 688 Attack Hub makes no formal claims
  about its own correctness. There is no frozen "level geometry" or
  geometric lemma to prove here — it's an idle/strategy toy, and pretending
  otherwise would be false rigor.
* **No accessibility audit.** The game has not been tested against WCAG
  criteria. If that matters to you, treat it as unaudited until someone
  actually runs one.
* **No performance/load claims.** One number exists — `positionMap()` costs
  0.34 ms per pass, measured once in one browser — and it is not a claim
  about anything else. The `step()` tick loop has never been profiled, no
  stress test exists at any scale beyond normal play, and nothing has been
  measured on a low-end device or a browser other than the one used.
* **No static analysis of the game itself.** CodeQL default setup is enabled
  and green, but it scans the `actions` language only: the two workflow
  files. It cannot reach the game's ~700 lines of inline JavaScript, because
  the repository detects as 100% HTML and CodeQL has no HTML extractor that
  would descend into a `<script>` block. Splitting the JS into a separate
  `.js` file would make it analysable, at the cost of the single-file
  property the rest of this repo is built around. That trade has not been
  made. Treat the game's JavaScript as unanalysed by any tool.
* **No security hardening claims beyond "makes no network calls."** See
  `SECURITY.md` for the deliberately small threat model this repo actually
  considers in scope.

=== Why the CI job is named `integrity`, not `verify`

Naming it `verify` would imply parity with f19-stealth-glider's verification
ledger, which this repo does not have. `integrity` names what the job
actually checks: that the repository's physical state matches what the docs
claim, not that any game-design property has been proven.
